    - Malformed detection is defined so that it partitions cleanly from cost_mismatch (c4) — no input class is claimed by both checks. A line is malformed iff: (a) it is not valid JSON, (b) it parses to a non-object (not a dict), or (c) it declares `kind` in {"session","artifact"} but lacks an identity/recompute key required to even classify and recompute the row — for a session row that key set is minimally `session_id`; for an artifact row it is `session_id`, `input_tokens`, and `output_tokens`. A missing or non-numeric stored `cost_*_usd` field (session) or stored `cost_usd` (artifact) is NOT malformed here — it is surfaced by c4 as a cost_mismatch (the recompute-vs-stored comparison treats an absent/non-numeric stored cost as a non-match). A row with NO `kind` field is NEVER malformed (it is classified legacy per c6). Malformed lines do not crash the audit; all subsequent lines are still processed and audited.
    - Cost-mismatch detection pins recompute targets per row kind. Session rows check stored `cost_llm_usd`, `cost_stt_usd`, `cost_tts_usd`, `cost_post_session_usd`, and `cost_total_usd` against the values recomputed from the row's own usage fields via the session recompute helper; artifact rows check stored `cost_usd` against the artifact recompute helper (from input/output tokens). A stored value that disagrees beyond a small rounding tolerance — OR is absent/non-numeric while a recomputed value exists — is flagged category "cost_mismatch" with line number and a reason naming the offending cost field and stored-vs-recomputed values. A fully-correct multi-component session row (all five components present and consistent) and a within-tolerance difference produce NO mismatch finding.
    - Orphan detection uses `session_id` as the linkage key via a whole-file (two-phase) pass: an `artifact` row is orphan iff no `session` row anywhere in the same ledger — before OR after it — shares its `session_id`, flagged category "orphan_artifact" with line number and reason. A correctly paired session+artifact (matching `session_id`) yields no orphan finding regardless of ordering. A session row with no matching artifact is NORMAL and never flagged.
    - Legacy rows are tolerated: legacy is defined STRICTLY as absence of the `kind` field. Such rows are classified legacy and skipped by every error check (never malformed, cost_mismatch, or orphan). Missing `mode` is NOT a legacy marker — a current session row (declaring `kind`:"session") that omits `mode` is a normal session row and MUST still be cost-checked.

Tests pass with API keys unset and pipecat absent, and the full existing suite stays green.
