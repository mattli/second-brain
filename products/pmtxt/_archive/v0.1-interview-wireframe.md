# pmtxt "The Interview" — Wireframe

Rough HTML wireframe for the V1 build target. Generated during the G-Stack `/office-hours` session on 2026-04-06 and directionally approved.

**HTML file:** `2026-04-06-pmtxt-interview-wireframe.html` (in this folder — open in a browser to view)

**Companion design doc:** `2026-04-06-pmtxt-interview-design.md`

## Three screens

### 1. The Interview
Conversational Q&A. Right side is the chat (AI questions, PM answers, AI pushback messages highlighted in amber when an answer is too vague). Left side is a dark sidebar listing the KB categories (customers, competitors, goals, constraints, decisions, revenue, market, principles) with depth indicators (rich / solid / sparse / empty) that fill in live as the PM answers questions. The PM watches their tacit product knowledge become structured in real time — that *is* the first moment of value.

### 2. Knowledge Base Overview
A card grid, one card per KB category, each with a depth badge (rich / solid / sparse / empty). Rich and solid cards summarize what the system extracted. Sparse cards show specific gaps ("Add growth targets to improve recommendations"). Empty cards say "optional — add when ready" rather than nagging. The PM sees their blind spots at a glance.

### 3. Recommendations
Prioritized feature ideas, each card showing: rank, title, rationale grounded in specific KB entries (not generic — references named customers and KB files), tags showing which KB sections informed the recommendation, and four actions: Copy Prompt (primary, generates a coding-agent-ready prompt), Approve, Defer, Deny. Low-confidence items (e.g. recommendations grounded in sparse KB sections) are visually muted with an explanation of what's missing.

## Key design decisions

- **The interview IS the product.** The PM gets value during onboarding, not after. No separate "setup" vs "use" phase.
- **Pushback is visible.** When the AI challenges a vague answer, that message gets a distinct amber treatment so the PM understands they're being pushed.
- **Depth is honest.** The four-tier indicator (empty / sparse / solid / rich) refuses to show fake percentages. It communicates confidence without false precision.
- **Recommendations reference the KB.** Each rationale cites specific files and customer names so the PM trusts the reasoning is grounded, not hallucinated.
- **"Copy Prompt" is the primary action.** The output flows directly into Claude Code / Cursor / etc. The PM doesn't need a PRD — they need a starting point for the coding agent.
