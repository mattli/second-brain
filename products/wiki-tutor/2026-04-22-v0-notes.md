# WikiTutor V0 — 2026-04-22 notes

## What I decided tonight

- Text-based, not voice. Voice had cost problems and voice-out wasn't the value anyway.
- Built for the Karpathy LLM Wiki audience — they have wikis and don't know what to do with them.
- Readwise integration is a V1 distribution play, not V0.
- V0 is a local web app that points at a directory of markdown files.
- Core product shape: tutor that teaches from a wiki page, with cross-page traversal via wikilinks, progress tracking per page, and a session wrap saved back to the vault.

## What got built (V0)

Next.js + TS + Tailwind, located at `/Users/mattli/Development/wiki-tutor/`. Four screens: Home, Session, Wrap, Coverage. Sample wiki with 13 pages. State in `.wikitutor/state.json`. Tutor logic is placeholder — real LLM calls drop into `src/lib/tutor.ts` later.

## What I noticed using it

- **Design feels too technical.** Fixable with UI polish. Not a product problem.
- **Questions overwhelmed me when I hadn't read the page.** This is the real insight. The tutor was asking me to engage with content I didn't have context on. The fix isn't cutting questions — it's adding a warmup phase where the tutor teaches first, then asks. "Teach first, question later."

## Open questions for tomorrow

- What does "teach first" actually look like in the session UI? A brief orientation paragraph before the questions load? An intro question like "want the tutor to walk you through this, or do you already know it?"
- Does the session structure become: overview → orientation Q → curriculum questions → wrap?
- How does the tutor decide how much to teach upfront vs. jump into questions? User config? Auto-detect based on coverage state?
- Does "teach first" also apply when the user branches to a linked page mid-session?
