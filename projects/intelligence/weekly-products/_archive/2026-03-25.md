# Product Briefing — 2026-03-25

## OpenAI Shuts Down Sora
**Who has this problem?** OpenAI's own product and video generation teams, plus Disney and the entertainment companies that bet on the platform
**What problem does it attempt to solve?** Sora launched in September 2025 as a TikTok-like social app built on top of the Sora 2 video model — the premise was an AI-first short-form video feed with "characters" (face-scan deepfakes) as its hook. It signed a $1B licensing deal with Disney in December 2025 to bring hundreds of IP characters into AI-generated video.
**Is the solution complete, or does it address only part of the pain?** The shutdown answers that: it doesn't work as a consumer product. Users made videos but didn't watch each other's — the novelty lasted weeks, not months. OpenAI is exiting not just the app but all public-facing video generation (consumer, developer API, and ChatGPT integration). The compute cost was crushing ahead of an IPO, and the content moderation problem (deepfakes, IP generation at scale) was unsolvable without destroying the product. Disney never closed the deal — no money changed hands. The Sora research team stays on to train world-simulation models for robotics.
**What would a better solution look like?** The honest answer may be that there is no consumer product here yet. AI video generation at social scale requires either a narrow enough use case (product demos, marketing clips) to avoid the moderation abyss, or a platform that controls discovery tightly enough to make watching AI content feel meaningful. The underlying model work continues behind the scenes — OpenAI is redirecting it toward physical-world robot training, which is the right long-term bet.

## Claude Code Auto Mode — AI That Decides What It Can Do
**Who has this problem?** Developers running long coding sessions in Claude Code who hit constant approval prompts — or who currently bypass all of them with `--dangerously-skip-permissions` and accept the risk
**What problem does it attempt to solve?** Anthropic introduced "auto mode," a new permissions tier where a classifier (running on Claude Sonnet 4.6) reviews every tool call before execution. Safe actions like reading files and writing to the working directory proceed automatically. Risky ones — mass file deletion, sensitive data exfiltration, running unknown binaries — get blocked and Claude is redirected to find a different path. If the same block keeps hitting, it escalates to a human prompt. It's a middle path between the default (constant friction) and skip-all-permissions (dangerous autonomy).
**Is the solution complete, or does it address only part of the pain?** Better than the binary that existed before, but the classifier's trust model needs configuration — it doesn't know your company's GitHub org or internal storage buckets by default, so those block until you whitelist them. Also adds some latency and token cost per tool call. Currently gated to Team plan users as a research preview.
**What would a better solution look like?** A classifier that bootstraps its trust model from project context (git remotes, `.claude/settings`, past session history) rather than requiring manual whitelist setup. The deeper question is whether the right design is approval-at-the-tool-call level at all, or whether agents should declare their intent upfront and get a scoped capability grant for a session — more like OAuth than a firewall.

## TurboQuant — Google's No-Loss LLM Compression
**Who has this problem?** ML infrastructure teams running large models at scale who face key-value cache memory as the primary bottleneck on context window length, throughput, and inference cost
**What problem does it attempt to solve?** Google Research published TurboQuant (ICLR 2026), a vector quantization algorithm that compresses the KV cache to 3–4 bits with mathematically provable zero accuracy loss. Unlike standard quantization that requires storing full-precision constants per block (adding 1–2 bits of overhead), TurboQuant eliminates this overhead entirely using two techniques: PolarQuant (rotation-based compression that maps to a fixed circular grid, removing the need for normalization) and QJL (a 1-bit residual error correction using the Johnson-Lindenstrauss transform). In benchmarks: 6x KV memory reduction, 8x attention compute speedup on H100 vs unquantized 32-bit, no degradation on LongBench/Needle-in-Haystack across Gemma and Mistral.
**Is the solution complete, or does it address only part of the pain?** Strong theoretical guarantees and clean benchmarks, but published by Google Research — not yet in a production inference stack you can pull from PyPI. The companion vector search results (near-optimal recall at 3-bit precision, no dataset-specific tuning required) are equally interesting for embedding-heavy workloads. The methodology is data-oblivious, meaning no per-model calibration.
**What would a better solution look like?** TurboQuant is close to optimal at what it does (it operates near theoretical lower bounds). The next layer is making it drop-in for existing inference stacks — vLLM, TensorRT-LLM, llama.cpp — so practitioners benefit without reading the ICLR paper. The vector search application is underreported: sub-3-bit recall at Google's ANN scale is a significant infrastructure story.

## VitruvianOS 0.3.0 — BeOS Lives Again on Linux
**Who has this problem?** Developers and technically curious users who find modern desktop Linux environments (GNOME, KDE) bloated, architecturally compromised by X11/Wayland's age, or simply designed around legacy assumptions about how humans use computers
**What problem does it attempt to solve?** VitruvianOS is a Linux-based desktop OS built from scratch to run the BeOS/Haiku API natively on a standard Linux kernel — without X Server or Wayland. The project ships its own graphical stack, a custom kernel subsystem called Nexus (analogous to Android's Binder) that bridges Linux with BeOS-style node monitoring and messaging, and real-time patched kernel for low-latency desktop use. The first public release, v0.3.0, landed March 23 after years of quiet development. The philosophy: human-first design, not feature accumulation. The Tracker (file manager), Deskbar, and core desktop components run as native Linux applications with BeOS API compatibility.
**Is the solution complete, or does it address only part of the pain?** Very early — v0.3.0 is explicitly a pilot/proof-of-concept. Filesystem indexing, live queries, and multi-user support are planned but not yet shipped. The real achievement is architectural: proving the BeOS runtime can run on Linux without forking X. BeOS veterans and Haiku contributors will find this immediately legible; everyone else gets a genuinely alien but coherent desktop experience.
**What would a better solution look like?** VitruvianOS is building the thing it claims. The gap is ecosystem — Haiku has a small but dedicated app library, and the question is whether the Linux kernel base (hardware support, driver coverage) combined with BeOS application compatibility is enough to attract developers who would otherwise just use Haiku or Nix. If the live query and filesystem indexing features land (BeOS's killer feature in 1995), this becomes interesting for data-heavy workflows.

## Video.js v10 — Four Open Source Projects Merge Into One
**Who has this problem?** Web developers embedding video who are currently duct-taping together a 700KB+ JavaScript player that was architected for a different era of the web, with UI customization requiring archeology into pseudo-element hacks
**What problem does it attempt to solve?** The original creator of Video.js reclaimed the project after 16 years and merged it with three other major open-source video players — Plyr, Vidstack, and Media Chrome — representing 75,000 combined GitHub stars and tens of billions of monthly video plays. The result: a ground-up rewrite that is 88% smaller (default bundle), ships first-class React/TypeScript support, introduces a composable feature system (Zustand-style slices), purpose-built presets for common use cases (video, audio, background), and a new streaming engine (SPF) that is 12% the size of HLS.js-light for simple adaptive bitrate use cases. The skin was redesigned by Sam Potts (creator of Plyr, known for its design quality).
**Is the solution complete, or does it address only part of the pain?** Beta, so rough edges remain — the "eject" workflow for custom skins is currently manual copy-paste, not a CLI. Multi-framework support beyond React and Web Components is roadmapped. For the common case (website video that's not TV-grade streaming), this is already meaningfully better than everything it's replacing.
**What would a better solution look like?** v10 is essentially already the better solution — the design decision to merge competing projects rather than fork again is the interesting move. The SPF streaming engine is the wildcard: if it matures to handle complex ABR cases (DRM, server-side ads, multi-CDN), it closes the last gap vs HLS.js and Shaka for production use. The AI-agent-friendly architecture note in the docs (explicitly designed so coding agents can build video players well) is a small tell about where the web development workflow is headed.
