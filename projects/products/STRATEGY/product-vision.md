# Product Vision

## North Star

Build something with positive impact on humanity or deep personal interest. No CRUD. Use what AI now makes possible that wasn't before.

---

## The Filter

Every new idea has to clear most of these. If it can't, it doesn't move forward.

1. **Do I live in this area?** Is the friction one I've personally felt, not just read about?
2. **Is someone paying for this outcome today?** A contractor, agency, subscription, or service I could displace.
3. **Can I sell the outcome, not the tool?** Autopilot, not co-pilot. The user sets a goal once; the system delivers results.
4. **Was this impossible or impractical before AI?** If a SaaS could've done it in 2020, skip.
5. **Can I build it as `loop + tools + prompts`?** No novel architecture required. Differentiation lives in the use case, prompts, and tools — not wiring cleverness.
6. **Is it narrow enough for a solo founder to go deep?** One vertical, real expertise, defensible by track record.

## Operating Beliefs

The reasoning behind the filter, in case I need to remember why.

**Services, not software.** *(Sequoia, "Services: The New Software", Mar 2026)* The valuable companies won't be tools — they'll be services that deliver outcomes. The moat is domain expertise, data, and trust in a specific vertical. Start with a person and a result, not a technical idea.

**Autopilot, not co-pilot.** Tools compete on features and price against other tools. Autopilots compete against what it costs to get the outcome another way — a contractor, agency, employee. Structurally higher value, structurally harder to replace. Tools get copied in weeks; outcomes require years of track record.

**LLM apps are loop + tools + prompts. Don't innovate over this.** Every successful LLM product is a model in a for-loop, calling tools, guided by prompts, until done. Because the architecture is commoditized, products scale with intelligence automatically — no rewrite when models improve. Clever architectures (complex multi-agent, novel memory, custom reasoning) get absorbed into base capabilities or turn out to be unnecessary. The wiki compiler is already this template: scheduled loop + Readwise/Filesystem MCP + instructions file as prompt.

**Personal friction beats sector analysis.** I live in most of my interest areas. The best ideas will come from things that have annoyed me, not from a TAM spreadsheet.
